# 0001: 認証・ID管理方式の選定

## ステータス

決定済み（2026-05-03）

## 背景

`tenforge.dev` ポータルおよび各サブドメインアプリ（`<appname>.tenforge.dev`）で共通利用する
認証・ID管理基盤が必要。以下の要件を満たす方式を選定する。

**要件:**
- 複数アプリ間でシングルサインオン（SSO）できること
- ユーザー登録・ログイン・セッション管理をセキュアに行えること
- 将来のマネタイズ（有料プラン管理）に対応できること
- 小規模スタートで運用コストを抑えられること

---

## 検討した選択肢

### 選択肢 A: Auth0（SaaS）

**概要:** Auth0 の無料/有料プランを利用し、認証を完全に外部委託する。

**メリット:**
- 実装コストがほぼゼロ（SDK を組み込むだけ）
- OAuth2 / OIDC / MFA / Social Login が即座に使える
- セキュリティ管理を Auth0 に任せられる

**デメリット:**
- 無料枠: MAU 7,500 人まで（超過後は高額）
- ユーザーデータが Auth0 側に保存される（ベンダーロック）
- カスタマイズに制約がある

---

### 選択肢 B: Keycloak（セルフホスト）

**概要:** OSS の Keycloak を VPS 上に Docker でセルフホストする。

**メリット:**
- 完全なコントロール（データ、カスタマイズ）
- MAU 課金なし
- OAuth2 / OIDC / SAML / Social Login 対応
- エンタープライズ機能（ロール管理、フェデレーション等）が無料

**デメリット:**
- 初期構築・運用コストが高い（JVM ベースでリソース消費大）
- VPS のスペック要件が高い（最低 1GB RAM 推奨、実用 2GB 以上）
- アップデート・メンテナンスが自己責任

---

### 選択肢 C: Supabase Auth（SaaS + OSS）

**概要:** Supabase の Auth 機能を利用する（PostgreSQL ベース）。

**メリット:**
- 無料枠: MAU 50,000 人まで（Auth0 より大きい）
- DB（PostgreSQL）も一緒に使えるため、ユーザーデータ管理が統合できる
- セルフホスト可能（将来の移行が容易）
- Row Level Security (RLS) でアプリ側のアクセス制御が書きやすい

**デメリット:**
- Supabase 全体への依存（Auth 単独分離が難しい）
- カスタム認証フロー（独自トークン形式等）は難しい

---

### 選択肢 D: Authentik（セルフホスト、軽量）

**概要:** Keycloak より軽量な OSS IdP。Python ベース。

**メリット:**
- Keycloak より低リソース（512MB RAM から動作）
- UI が現代的でわかりやすい
- OAuth2 / OIDC / SAML / Social Login 対応
- Docker Compose で簡単に起動できる

**デメリット:**
- Keycloak ほどの実績・コミュニティがない
- エンタープライズ機能は有料（Enterprise tier）
- 日本語ドキュメントが少ない

---

### 選択肢 E: 自前実装（JWT + セッション）

**概要:** FastAPI + JWT でスクラッチ実装する。

**メリット:**
- 完全なコントロール
- 依存ゼロ、シンプル

**デメリット:**
- セキュリティリスクが高い（実装ミスが脆弱性に直結）
- OAuth2 Social Login の実装コストが大きい
- MFA・パスワードリセットなど周辺機能を全部作る必要がある
- **推奨しない**（認証はよほどの理由がない限り自前実装すべきでない）

---

## VPS 調査結果（2026-05-03 時点）

| 項目 | 値 |
|---|---|
| OS | Ubuntu 24.04.3 LTS |
| RAM | 1.9GB（通常稼働時 ~1.3GB 空き） |
| CPU | 3コア |
| Disk | 99GB（空き 85GB） |
| 稼働中アプリ | kakeibo.tenforge.dev（FastAPI + Docker + nginx + SSL） |

Keycloak は JVM で 1GB+ 消費するためスペック的に不適。
Authentik は通常時なら動くが余裕が薄く、将来のアプリ追加を考えると採用リスクが高い。

---

## 決定

**選択肢 C: Supabase Auth** を採用する。

**マネタイズ連携: Stripe** を採用する。

---

## 理由

### Supabase Auth を選ぶ理由

1. **VPS の RAM を消費しない** — 認証処理が Supabase のクラウド上で完結する
2. **MAU 50,000 まで無料** — スケールするまで課金なし
3. **PostgreSQL 込み** — ユーザーデータ・アプリデータを同じ DB で管理できる
4. **RLS（Row Level Security）** — アプリ側のアクセス制御が SQL ポリシーで書ける
5. **Stripe との公式連携テンプレートあり** — マネタイズ実装が速い
6. **セルフホスト移行パスあり** — ユーザー増加後に VPS 移行可能

### Stripe を選ぶ理由

1. **業界標準** — ドキュメント・事例が最も豊富
2. **Supabase との相性が良い** — Webhook → Supabase Edge Functions の連携が定番
3. **機能の網羅性** — サブスクリプション / 一回払い / 請求ポータル / 税務 / インボイスをすべてカバー
4. **日本円対応** — JPY での決済・振込に対応

### マネタイズアーキテクチャ（推奨構成）

```
ユーザー → Supabase Auth（ログイン）
              ↓
         Stripe Customer 紐付け（user_id ↔ stripe_customer_id）
              ↓
         Stripe Checkout でサブスクリプション購入
              ↓
         Webhook → Supabase Edge Function → DB の plan カラム更新
              ↓
         アプリ側: RLS ポリシーで plan に応じた機能制限
```

---

## トレードオフ

| | Supabase Auth + Stripe | Authentik（セルフホスト） |
|---|---|---|
| 初期構築コスト | 低 | 中 |
| VPS RAM 消費 | なし | ~700MB |
| MAU 無料枠 | 50,000 | 無制限 |
| マネタイズ連携 | Stripe と公式連携あり | 自前実装が必要 |
| データ主権 | Supabase 管理（移行可） | 完全自己管理 |
| 将来の移行 | セルフホスト Supabase へ移行可 | — |

**失うもの:** ユーザーデータが一時的に Supabase 管理下に置かれる。ただしエクスポートは常に可能で、セルフホスト移行パスがある。
